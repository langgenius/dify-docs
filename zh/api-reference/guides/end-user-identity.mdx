大多数应用接口都接受一个 `user` 字段：由你自己定义的标识符，表示这次调用代表哪位终端用户。Dify 不会对它做任何认证，为每个人选一个稳定的值（如账户 ID），并在每次调用中发送同一个值。

新的 `user` 第一次出现时，Dify 就会开始跟踪它，一个应用密钥可服务任意数量的终端用户。

## `user` 的作用范围

`user` 决定每次调用能看到什么、能做什么：

- **会话**：列表、历史、重命名和删除都只作用于该用户自己的会话。
- **文件**：上传的文件归上传者所有，换一个 `user` 引用该文件会失败。
- **停止生成**：对话类应用的停止接口仅在 `user` 与发起回复的用户一致时生效。不一致时会被静默忽略：调用同样会成功，因此无法从响应中察觉。

    Workflow 应用的「停止工作流任务」则不检查 `user`。

- **恢复运行**：`user` 与运行创建者不一致时，[流式获取工作流事件](/zh/api-reference/workflow-runs/stream-workflow-events) 会返回 404。

同一个人在流程的每次调用中都使用同一个 `user`：上传、发送、停止、恢复都会检查它。

## API 用户与 WebApp 用户相互独立

API 流量与 Dify 托管的 WebApp 使用彼此独立的身份：API 用户创建的会话不会出现在 WebApp 中，WebApp 中的会话也不会出现在 API 里。

## 解析终端用户 ID

有些响应只携带终端用户 ID 而非完整信息，例如上传文件响应中的 `created_by`。可用 [获取终端用户信息](/zh/api-reference/end-users/get-end-user-info) 解析它。
