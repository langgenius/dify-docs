After you build and publish a Workflow app, you can deploy it to one or more runtime environments from **Deployments** in the workspace.

Each environment runs the app separately, with its own release and access settings. Your end users can keep using a stable release while you test a new one somewhere else.

<Info>
Currently, only Workflow apps can be deployed. Some workflow nodes aren't supported yet; if your app uses one, the deployment flow flags it before you proceed.
</Info>

## How Deployments Work

To ship an app, you create a deployment.

During creation, you select a published Workflow app and build a release from it. That release can then be deployed to a target environment, where the app runs as an instance.

As you keep publishing new versions in **Studio**, you can build more releases in the same deployment and deploy them.

A release can run in several environments at once, and you can switch the release an environment runs at any time: deploy a newer one, or roll back to an earlier one.

For example, in a `customer-support` deployment, you might run release `v1.2` in `production` while testing `v1.3` in `staging`. Once `v1.3` is verified, you can deploy it to `production` too.

## Key Concepts

### Deployment

A deployment holds your releases, the instances running them across environments, and their access settings.

You usually create one deployment per app, while several deployments can exist for the same app when separate teams need to ship the app and manage releases independently.

### Release

A release is a frozen snapshot of your app, built from its latest published version in **Studio**. Once created, a release stays available to deploy unless you delete it.

Different releases can be deployed to different environments, and the same release can be deployed to several environments at the same time.

### Environment

An environment is a runtime location for a deployed release, such as `testing`, `staging`, or `production`.

Environments are set up by administrators from the management console, outside your workspace, so you deploy to the ones available to you. If you need one that isn't listed, ask your administrator.

Each environment can run a release using its own configuration, including model and plugin credentials, along with any environment variable values the release needs. For example, you can use separate model API keys for `testing` and `production`.

<Tip>
A built-in `system` environment is always available for trying things out.
</Tip>

### Instance

An instance is the running app your end users reach in an environment.

Instances are environment-specific and each run one release at a time. You upgrade or roll back an instance by deploying a different release to the environment where the instance runs.
