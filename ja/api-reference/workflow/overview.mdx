Workflow アプリは `workflow` モードで動作します。呼び出しごとに公開済みのワークフローを 1 回実行し、その出力を返します。呼び出し間で会話状態を保持しないため、各実行は独立しています。

## ワークフローの実行

[ワークフローを実行](/ja/api-reference/workflow-runs/run-workflow) は、指定した入力変数でアプリの現在の公開バージョンを実行します。[ID でワークフローを実行](/ja/api-reference/workflow-runs/run-workflow-by-id) は、特定の公開バージョンを実行します。`workflow_id` は、以前の実行結果や実行詳細のレスポンスから取得できます。

## 実行の追跡と制御

ワークフロー実行時に `response_mode` を `streaming` にすると、実行中のイベントをリアルタイムに取得できます。`blocking` にすると、完了まで待機してから結果を受け取ります。どちらのモードでも、レスポンスには 2 つの ID が含まれます。`task_id` は実行中のタスクを制御し、`workflow_run_id` は実行後に保存される記録を識別します。

[ワークフロータスクを停止](/ja/api-reference/workflow-runs/stop-workflow-task) は、`task_id` を指定して実行中のタスクを取り消します。ただし `streaming` モードでのみ有効です。`blocking` モードでは、レスポンスが返る時点ですでに実行が完了しているため取り消せません。

接続が切断された場合や実行が一時停止した場合は、[ワークフローイベントをストリーム](/ja/api-reference/workflow-runs/stream-workflow-events) に `workflow_run_id` を渡してストリームを再開できます。`include_state_snapshot=true` を指定すると、実行済みノードの状態を先に再生します。`continue_on_pause=true` を指定すると、複数回の人間の入力による一時停止をまたいでストリームを維持できます。

## 人間の入力による一時停止

実行が人間の入力ノードで一時停止すると、ストリームに `form_token` を含む `human_input_required` イベントが発生します。このイベントは `workflow_paused` と同時に発生し、その後ストリームは閉じます。ノードの配信方法が WebApp の場合は、[人間の入力フォームを取得](/ja/api-reference/human-input/get-human-input-form) でフォームの内容を取得し、[人間の入力フォームを送信](/ja/api-reference/human-input/submit-human-input-form) で回答者の入力内容を送信します。

フォームを送信すると、実行が再開します。回答者が時間内に応答しない場合は、設定されていればタイムアウト後のフォールバックエッジに従って実行が進みます。いずれの場合も、一時停止前のストリームに含まれる `workflow_run_id` を使います。[ワークフローイベントをストリーム](/ja/api-reference/workflow-runs/stream-workflow-events) で残りのイベントを追跡できます。

## 実行履歴の確認

[ワークフロー実行詳細を取得](/ja/api-reference/workflow-runs/get-workflow-run-detail) は、指定した `workflow_run_id` の実行ステータスと出力を返します。[ワークフローログ一覧を取得](/ja/api-reference/workflow-runs/list-workflow-logs) が返すのは、ステータス、token 使用量、ステップ数、所要時間といった実行単位のサマリーです。ノード単位の実行ログは含まれません。キーワード、ステータス、期間、または実行を開始したエンドユーザーで絞り込めます。

ノード単位の詳細が必要な場合は、ログの一覧取得ではなく実行自体をストリームしてください。開始する実行には [ワークフローを実行](/ja/api-reference/workflow-runs/run-workflow) で `response_mode=streaming` を指定します。すでに進行中の実行には [ワークフローイベントをストリーム](/ja/api-reference/workflow-runs/stream-workflow-events) を使用します。

## ファイルの操作

[ファイルをアップロード](/ja/api-reference/files/upload-file) は、画像、ドキュメント、音声、動画ファイルをアップロードし、ワークフローの実行から参照できるようにします。[ワークフローを実行](/ja/api-reference/workflow-runs/run-workflow) や [ID でワークフローを実行](/ja/api-reference/workflow-runs/run-workflow-by-id) を呼び出す際は、ファイル型の入力変数を使います。値には、レスポンスで返された `id` を `upload_file_id` として渡します。Workflow アプリには、アップロード後のファイルをプレビューまたはダウンロードするエンドポイントはありません。

アップロードファイルのレスポンスの `created_by` のように、レスポンスにはエンドユーザー ID のみが含まれることがあります。その場合は、[エンドユーザー取得](/ja/api-reference/end-users/get-end-user-info) で詳細情報を取得できます。

## 音声とテキストの相互変換

[音声をテキストに変換](/ja/api-reference/audio/convert-audio-to-text) は、アップロードされた音声ファイル（MP3、M4A、WAV、AMR、MPGA のいずれか、最大 30 MB）をテキストに変換します。エンドユーザーは入力値をテキストではなく音声で入力できます。[テキストを音声に変換](/ja/api-reference/audio/convert-text-to-audio) は、ワークフローのテキスト出力を音声として合成します。

## アプリ情報と設定の取得

[アプリケーションの基本情報を取得](/ja/api-reference/applications/get-app-info) は、アプリの名前、説明、タグ、モードを返します。[アプリケーションのパラメータ情報を取得](/ja/api-reference/applications/get-app-parameters) は、機能スイッチや入力パラメータの名前・型・デフォルト値といった入力フォームの設定を返します。クライアント UI の構築に役立ちます。

[アプリケーションのメタ情報を取得](/ja/api-reference/applications/get-app-meta) は、ツールアイコンなどの設定メタデータを返します。[アプリの WebApp 設定を取得](/ja/api-reference/applications/get-app-webapp-settings) は、WebApp のサイト設定、テーマ、カスタマイズ項目を返します。
